<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Rails app | User submits notes form data (body, file_purge, file) — markdown body later rendered as HTML |
| Stimulus controller → window.location | Row-click navigates to a server-derived URL embedded in a data attribute |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | row_link_controller.js reading `data-row-link` | mitigate | The data attribute value is server-rendered via `note_path(note)` (Rails URL helper, escaped). Controller does not concatenate user input. No `eval` or `innerHTML`. |
| T-03-02 | XSS via markdown body | show.html.erb render_markdown sanitize call (existing) | accept | Sanitization handled in Phase 2 (D-02). No new render path introduced; show page wrapping change is structural only. |
| T-03-03 | Clickjacking | New top-right action bar (Edit/Delete) | accept (deferred to Plan 02) | Plan 03-02 adds `X-Frame-Options: SAMEORIGIN` globally per D-SEC-03 — no per-page mitigation needed here. |
| T-03-04 | Repudiation | Delete-from-table button | mitigate | `turbo_confirm` requires explicit user acceptance before issuing DELETE; existing Devise auth + CSRF token (Rails default) bind action to current_user. |
| T-03-05 | Information disclosure | Per-field error rendering exposing model state | accept | `note.errors[:body].full_messages` already public Rails behavior; no PII or encrypted body leaked — only validation strings. |
</threat_model>

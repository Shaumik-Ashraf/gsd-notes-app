<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Stored body -> rendered HTML | A note body is user-controlled markdown. After encryption-at-rest, it is decrypted in-process and converted to HTML. Sanitization is the boundary that prevents XSS. |
| URL :id parameter -> note row | Direct navigation to /notes/:id can target any integer; scoping is the only thing preventing horizontal access. |
| Active Storage blob URL -> file disk | The signed blob URL provides time-limited authenticated download. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Tampering | Stored XSS via markdown body | mitigate | `sanitize(render_markdown(@note.body), tags: %w[p h1 h2 h3 h4 h5 h6 ul ol li a code pre blockquote strong em br hr], attributes: %w[href])` (D-02). The sanitize call wraps the render output — not vice versa. Tested by sanitization-system-test in Task 1 plus the inline `bin/rails runner` acceptance criterion. |
| T-02-03-02 | Information Disclosure | Cross-user GET /notes/:id | mitigate | `set_note` uses `current_user.notes.find(params[:id])`; non-owned ID raises `ActiveRecord::RecordNotFound` -> 404. System test "show page returns 404 when accessing another user's note" proves it. |
| T-02-03-03 | Information Disclosure | Direct Active Storage blob URL guessing | mitigate | `rails_blob_path` generates signed URLs (Rails 8 default ActiveStorage uses signed `/rails/active_storage/blobs/:signed_id/...`). Unauthenticated GETs to the signed URL are not blocked, but the signed_id is not enumerable. Production hardening to fully authenticate blob downloads is Phase 3 scope. |
| T-02-03-04 | Tampering | `javascript:` URI in markdown link href | mitigate | Rails `sanitize` strips `javascript:` href by default in the allowlist sanitizer; `attributes: %w[href]` only permits the attribute, not arbitrary scheme. Tested implicitly by sanitize defaults. |
| T-02-03-05 | Information Disclosure | Attachment filename in URL reveals existence | accept | Filenames embedded in the signed URL are visible to anyone with the link; this is standard Rails Active Storage behavior. Users only share links of their own notes. |
| T-02-03-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan. |
</threat_model>

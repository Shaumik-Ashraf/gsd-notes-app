<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser PATCH/DELETE -> NotesController | Edit and delete are state-changing; must be authenticated, CSRF-protected, scoped, and idempotent on the auth check. |
| `file_purge` param -> Active Storage | A truthy `note[file_purge]=1` triggers attachment destruction. If mass-assignment misroutes, an attacker could purge attachments they should not be able to. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-01 | Spoofing | Edit/Update/Destroy as another user | mitigate | `set_note` uses `current_user.notes.find(...)`; cross-user IDs raise `RecordNotFound` -> 404. Tested by cross-user edit + destroy system tests in Task 1. |
| T-02-04-02 | Tampering | Mass-assignment escalation | mitigate | `note_params` permits ONLY `:body, :file, :file_purge` (set in Plan 02). `:user_id`, `:created_at` etc. cannot be reassigned via update. |
| T-02-04-03 | Information Disclosure | Old attachment blob remains after replace | mitigate | Active Storage's `has_one_attached` replaces the attachment record; Plan 01's `dependent: :purge` ensures the old blob is purged on note destroy. For replace mid-life, Active Storage's `attach` on a `has_one_attached` auto-purges the prior attachment in Rails 8. Tested by the "edit replaces the attachment" system test which asserts the old filename no longer renders. |
| T-02-04-04 | Denial of Service | Synchronous purge on large file | accept | Per D-07, synchronous purge is intentional for v1 scale. Large-file purge latency is acceptable. |
| T-02-04-05 | Tampering | CSRF on PATCH/DELETE | mitigate | Rails default CSRF protection; Capybara form submits include the token. `button_to ... method: :delete` issues a POST with `_method=delete` and CSRF token ŌĆö the system test exercises this path. |
| T-02-04-06 | Repudiation | Delete without confirmation | mitigate | `data: { turbo_confirm: "Delete this note? This cannot be undone." }` on the destroy button (from Plan 03 show view + index from Plan 02). System test uses `accept_confirm` to exercise the dialog. |
| T-02-04-07 | Information Disclosure | Body cleared but file_purge=true with no replacement = note becomes invalid mid-edit | mitigate | NOTE-10 validation rejects the update; the existing body and attachment are preserved server-side because `update` returns false and renders :edit. Tested by "edit with body cleared and attachment removed" system test. |
| T-02-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan. |
</threat_model>

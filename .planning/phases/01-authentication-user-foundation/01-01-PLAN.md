<threat_model>
## Threat Model (ASVS L1)

**Trust boundary:** unauthenticated browser → Rails (Devise) → SQLite users table.

| Threat ID | Category | Component | Severity | Disposition | Mitigation |
|-----------|----------|-----------|----------|-------------|------------|
| T-01-01 | Spoofing | User sign-in path | High | mitigate | Devise `:database_authenticatable` uses bcrypt-hashed passwords (cost factor default 12); `encrypted_password` column never exposed in any view or log (param filter `:passw` from CLAUDE.md covers `password`) |
| T-01-02 | Elevation of Privilege | Self-registration route | High | mitigate | `devise_for :users, skip: [:registrations]` removes all registration routes; verified via `bin/rails routes` not listing `new_user_registration` (acceptance criterion in Task 3) |
| T-01-03 | Elevation of Privilege | Controller skipping authenticate_user! | High | mitigate | Architectural invariant D-12/D-13: all authenticated controllers subclass `AuthenticatedController`. RootController demonstrates the pattern; checker plan will grep for any controller inheriting directly from ApplicationController without justification |
| T-01-04 | Information Disclosure | Generic sign-in error | Low | mitigate | D-06 — error message "Invalid email or password." does not distinguish unknown email vs. wrong password (Plan 02 enforces in view) |
| T-01-05 | Information Disclosure | Devise default mailer placeholder | Low | accept | `:recoverable` is enabled but ActionMailer delivery is not configured (D-04). Recovery emails are not sent in Phase 1. Webmaster handles resets via console. Documented as production prerequisite in CONTEXT.md Deferred section |
| T-01-SC | Tampering | npm/gem supply chain — `devise` install | Medium | mitigate | `devise` is a well-known gem (heartcombo/devise, >70M downloads). No `[ASSUMED]`/`[SUS]` packages; no slopcheck block required. Bundler-audit (already in Gemfile dev/test) flags CVEs |
| T-01-06 | Tampering | `:rememberable` re-introduction | Medium | mitigate | D-11 explicit: rememberable removed from model AND migration. Acceptance criteria grep for `:rememberable` and `remember_created_at` returning zero matches |

**Security notes:**
- `/up` health check is intentionally public (D-14) — no PII or app state leaks; required for Kamal/load balancer liveness.
- Session cookie defaults: Rails 8.1 ships with `same_site: :lax` and `secure` in production (config/environments/production.rb default) — no change needed in Phase 1.
- CSRF: ApplicationController inherits `ActionController::Base` which enables CSRF by default — no change needed.
- No password complexity policy added in Phase 1 (Devise default minimum 6 chars via `:validatable`). Hardening to ASVS L1 password rules is Phase 3 polish scope.
</threat_model>

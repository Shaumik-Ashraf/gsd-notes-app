## Threat Model (ASVS L1)

**Trust boundary:** unauthenticated browser → Devise sessions controller → User record.

| Threat ID | Category | Component | Severity | Disposition | Mitigation |
|-----------|----------|-----------|----------|-------------|------------|
| T-01-07 | Information Disclosure | Sign-in error message | Medium | mitigate | D-06 — the literal copy is `Invalid email or password.` (Task 1 edits `config/locales/devise.en.yml`; Task 3 asserts the exact string). No distinction between unknown email and wrong password — prevents account enumeration |
| T-01-08 | Elevation of Privilege | Stale `_links.html.erb` partial | Medium | mitigate | Task 1 neutralizes `app/views/devise/shared/_links.html.erb` to an ERB comment so no Devise view (now or in the future) accidentally renders a sign-up or forgot-password link |
| T-01-09 | Spoofing | Session fixation across sign-out | High | mitigate | Devise's default sign-out path destroys the session (`reset_session`). Task 3 system test asserts that after sign-out, `root_path` redirects to `new_user_session_path` — proving the session is no longer valid |
| T-01-10 | Tampering | CSRF on sign-in POST | High | mitigate | Inherited from `ActionController::Base` (CSRF protection default-on). Devise's `form_for` emits the CSRF token automatically. No change needed |
| T-01-11 | Information Disclosure | Password autofill | Low | accept | `autocomplete="current-password"` is intentional (browsers' password manager UX). ASVS L1 does not prohibit |
| T-01-12 | Denial of Service | Brute-force login | Medium | accept | No rate-limit / lockable module in Phase 1 (D-02 frozen module list excludes `:lockable`). Documented for Phase 3 hardening |
| T-01-13 | Information Disclosure | Forgot-password link without mailer | Medium | mitigate | UI-SPEC + Task 1 remove the "Forgot your password?" link from the sign-in view. Devise `new_user_password` route still exists at the controller level but is unlinked in Phase 1 — visiting it directly would produce a mailer error; this is acceptable because the link is not exposed in any UI and webmaster owns recovery via console |

**Security notes:**
- No new packages installed in this plan — only view, CSS, controller-override, and test changes. Package legitimacy gate not applicable.
- Test isolation: system tests use a transactional database fixture; `User.create!` in setup is rolled back per test.
- Headless Chrome driver default is fine; no need for network sandbox config in Phase 1.
- The `:recoverable` route remaining at the controller level (without mailer) is a known accept (T-01-05 from Plan 01-01). Phase 3 owns the production prerequisite.
</threat_model>
